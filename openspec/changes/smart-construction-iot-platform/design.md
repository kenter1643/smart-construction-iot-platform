## 背景

智慧工地物联网平台的建立旨在解决建筑行业在安全、效率和实时监控方面的挑战。主要利益相关者包括施工现场经理、安全检查员和设备操作员。该平台必须能够使用通用的通讯协议与各种物联网设备（传感器、摄像头、机械）进行集成。

### 当前状态
- 大多数建筑工地依赖人工检查和纸质记录
- 数据通常是孤立的，无法实时访问
- 传统监控方法耗时且容易出错

### 限制条件
- 必须支持 Modbus TCP、MQTT 和 HTTP 通讯协议
- 必须处理高数据吞吐量和大量设备
- 必须具有可扩展性，以支持多个建筑工地
- 必须符合相关的安全和隐私法规

## 目标/非目标

**目标：**
- 提供设备和传感器数据的实时可视化
- 实现通过 Modbus TCP、MQTT 和 HTTP 的设备通讯
- 支持视频流接入和转发
- 实现设备管理功能
- 提供异常情况的告警和通知
- 确保平台具有可扩展性、可靠性和安全性

**非目标：**
- 构建物理硬件设备（重点是平台软件）
- 支持所有可能的通讯协议（最初仅支持 Modbus TCP、MQTT、HTTP）
- 替换现有的企业资源规划（ERP）系统

## 设计决策

### 架构风格
**决策：** 使用微服务架构
**理由：** 微服务允许不同平台组件（如设备通讯、视频流、数据存储）的独立扩展和部署。这使得平台更具弹性，并能适应未来的变化。

### 通讯协议
**决策：**
- 使用 MQTT 处理高容量传感器数据
- 使用 Modbus TCP 进行工业机械通讯
- 使用 HTTP 进行设备配置和管理
**理由：** MQTT 对于低功耗、高吞吐量的传感器数据效率很高。Modbus TCP 在工业自动化中被广泛采用。HTTP 易于与现有工具和 API 集成。

### 消息代理
**决策：** 使用带 MQTT 插件的 RabbitMQ
**理由：** RabbitMQ 支持 MQTT 和 AMQP，为未来的协议支持提供了灵活性。MQTT 插件简化了与物联网设备的集成。

### 视频流
**决策：** 使用 RTSP 进行视频摄入，HLS 用于网页播放
**理由：** RTSP 是 IP 摄像机视频流的标准。HLS 被网页浏览器广泛支持。

### 视频查看功能
**决策：** 使用 HTML5 Video 和 React Player 库
**理由：** HTML5 Video 是现代浏览器原生支持的视频播放技术。React Player 是一个强大的 React 视频播放器库，支持多种格式（包括 HLS），并提供播放控制、全屏等功能。

### 视频存储格式
**决策：** 使用 MP4 格式存储录制的视频
**理由：** MP4 是一种广泛支持的视频格式，与各种播放器兼容，并且文件大小适中。

### 数据存储
**决策：** 使用 InfluxDB 存储时间序列数据，PostgreSQL 存储关系数据
**理由：** InfluxDB 针对时间序列数据进行了优化，非常适合传感器数据。PostgreSQL 是用于设备元数据和用户信息的强大关系型数据库。

### 实时处理
**决策：** 使用 Apache Kafka 进行实时数据流处理
**理由：** Kafka 提供高吞吐量和低延迟的实时数据处理，非常适合处理大量传感器数据。

### 仪表盘
**决策：** 使用 React 与 D3.js 和 Socket.IO
**理由：** React 是用于构建交互式 UI 的流行 JavaScript 库。D3.js 提供强大的数据可视化功能。Socket.IO 实现服务器和客户端之间的实时通讯。

## 风险/权衡

### 风险：高设备连接量
**风险：** 平台可能难以处理大量并发设备连接
**缓解：** 为消息代理实现水平扩展，并使用负载均衡来分配流量

### 风险：数据延迟
**风险：** 实时数据可能因高流量或处理瓶颈而延迟
**缓解：** 使用 Kafka 进行高吞吐量数据流处理，并优化数据处理管道

### 风险：安全漏洞
**风险：** 平台可能容易受到攻击（如设备伪造、数据泄露）
**缓解：** 实施强大的身份认证、数据加密和定期安全审计

### 风险：设备兼容性
**风险：** 某些设备可能不支持受支持的通讯协议
**缓解：** 提供协议转换服务和设备 SDK，以简化集成
